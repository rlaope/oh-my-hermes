Why OMH exists

The fit layer between plain Hermes requests and trustworthy work.

OMH is not another dashboard to learn. It gives Hermes deterministic local contracts for routing, status, planning, capabilities, plugin observations, and coding handoffs while the user keeps speaking in normal chat.

Modern OMH is a contract system, not a command catalog. Wrapper surfaces can render omh_interact, chat_response/v1, and coding_runtime_handoff/v1 without turning a prepared next step into observed work.

01

Classify the request

Natural messages become responsible roles, workflow lanes, and first safe actions without asking users to memorize command names.

02

Prepare the contract

Plans, status cards, capability manifests, and handoff payloads are explicit artifacts that wrappers can render.

03

Observe before claiming

Execution, review, CI, delivery, plugin load, and merge readiness stay gated until a local record proves them.

Evidence boundary

Architecture at a glance.

Hermes owns chat intake, clarification, research, planning, and narration. OMH owns the renderable contract layer. Selected executors and runtime artifacts own observed proof.

01

Message classified

Route, role, language, and first action are available.

02

Plan accepted

03

Handoff prepared

The selected executor can receive the work, but no work is claimed yet.

04

Run observed

Runtime records carry implementation, verification, review, and CI evidence.

05

Status rendered

Wrappers show the strongest true state and the remaining gates.

Wrapper-ready by design

Contracts that adapters can render without scraping prose.

Discord, Slack, hosted chat, and Hermes-native plugin surfaces can consume local envelopes instead of guessing what a workflow means. The UI can show actions, disabled gates, and evidence labels from the same state. Adapter metadata can still carry harness_progress/v1 when a wrapper needs the raw contract.
