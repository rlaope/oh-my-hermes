Core values

01Install once

Set up the local OMH surface without making users learn a new dashboard.

02Keep Hermes

Hermes owns chat intake, clarification, research, planning, and narration.

03Smallest safe move

Every natural request gets a visible route, role, and next action.

04Local contracts

Contracts, fixtures, status, and capability data stay deterministic and inspectable.

05Proof boundary

Prepared handoff is not execution evidence.

Chat wrappers

Wrappers own platform auth, events, buttons, threads, edits, and notifications. Hermes owns the chat experience. OMH returns the renderable contract.

plugin tool: omh_interact omh chat interact --source discord "risky refactor" omh chat route --source discord --summary "risky refactor"
Wrapper rule

Render chat_response/v1 and action state. Do not ask users to memorize backend commands.

Role surface

Roles name responsibility for Hermes chat and wrapper UI. They do not claim that a hidden worker ran.

guide

First-touch routing and route explanation.

researcher

Source-backed discovery and unknowns.

planner

Clarification, non-goals, and plan acceptance.

handoff-guide

Executor choice and prepared handoff payloads.

Delegation-first coding

  • OMH does not hide a coding executor inside Hermes.
  • Prepared handoff means "ready to dispatch", not "work completed".
  • Observed implementation, review, CI, and merge readiness require explicit records.

Boundaries

OMH is deterministic local tooling. It does not patch Hermes core, run platform bots, call LLM APIs, create network integrations inside core commands, or launch coding executors by itself.
