Architecture guide

Hermes Agent is not just a chat box. It is a work operating layer.

Hermes Agent combines a terminal UI, messenger gateway, memory, skills, cron jobs, tools, plugins, platform adapters, and agent loops. OMH should sit on top as a workflow pack and evidence layer, not as a hidden runtime replacement.

Surfaces Terminal, CLI, gateway, Telegram, Discord, Slack, WhatsApp, Signal
Core loop AIAgent, prompt builder, model transport, tool loop, session persistence
Context brain Memory manager, session search, context files, compression, user model
Workflow body Skills, skill creation, curator, cron, kanban, profiles, subagents
Extension ports Tools, toolsets, MCP, plugins, platform registry, terminal backends
OMH layer Installable workflows, routing, handoffs, status cards, evidence contracts

The simple mental model

A normal chatbot answers one conversation. Hermes Agent is built closer to an operating surface for work. It can be entered from the terminal or a messaging gateway. It builds a prompt from identity, context, skills, memory, and platform hints. It can call tools, run scheduled jobs, spawn delegated work, persist session history, and use skills as procedural memory.

That is why OMH should not sell itself as "more commands." The product value is making those Hermes capabilities easier to start, safer to inspect, and clearer to explain in chat.

A request does not just become an answer. It becomes a route.

The most useful way to understand Hermes is to follow one message. A user can ask from a terminal, Discord, Slack, Telegram, or another gateway surface. Hermes builds context, loads relevant skills, decides whether memory or tools matter, then returns a user-facing response. OMH adds the missing product shape: what workflow was chosen, what the next action is, and what still has not been observed.

1

Message arrives

CLI, TUI, or gateway session provides platform, thread, and user context.

2

Prompt is built

Identity, skills, memory guidance, context files, platform hints, and safety checks are assembled.

3

Tools may run

Hermes can call allowed tools, search sessions, use memory, or prepare scheduled work.

4

OMH names the workflow

The response can show plan, research, feedback triage, loop, image card, or coding handoff.

5

Evidence advances later

Status moves only when wrapper, executor, CI, delivery, or review evidence is recorded.

01

Conversation surface

Hermes exposes both terminal UX and gateway UX. The gateway registry lets built-in or plugin platform adapters register auth, setup, delivery, privacy, and cron delivery behavior.

02

Agent loop

run_agent.py hosts AIAgent, session transitions, context engine lifecycle, tool calls, response handling, persistence, and background review hooks.

03

Memory and context

agent/memory_manager.py coordinates built-in memory plus at most one external provider, prefetch, sync, and streaming scrubbing so recalled context does not leak into final output.

04

Skills and learning

The prompt builder tells Hermes when to save or patch skills. The curator keeps agent-created skills maintainable. This gives Hermes a learning-friendly loop without requiring OMH to train a model.

05

Cron and work orchestration

cron/scheduler.py uses locks, protected toolset policy, delivery targets, parallel pools, and prompt-injection scanning to run scheduled work with guardrails.

06

OMH contract layer

OMH gives Hermes workflow recipes, deterministic routing, status cards, coding handoff contracts, and prepared-vs-observed boundaries so wrappers can show what happened without guessing.

Myths and reality

Myth: OMH should become the executor.

Reality: Hermes and selected executors already own runtime work. OMH should make the route, handoff, status, and evidence boundary visible.

Myth: more commands equals a stronger product.

Reality: normal users talk to Hermes. Commands are installer, verifier, backend, and operator infrastructure.

Myth: a plan is progress evidence.

Reality: a plan is prepared work. Dispatch, implementation, review, CI, and delivery need separate observations.

Myth: self-improvement means automatic mutation.

Reality: the safer loop is trace, evaluate, propose, review, patch, and replay. Human review keeps the skill base from drifting.

Where OMH fits

User asks Hermes "Can you make this safer, clearer, or ready to ship?"
Hermes thinks and talks Clarify, research, summarize, plan, and narrate status.
OMH shapes the workflow Pick a workflow, prepare a handoff, record local evidence state.
Executor or wrapper observes Dispatch, result, review, CI, delivery, or merge evidence.

The key rule is still conservative: prepared is not observed. A plan, route, prompt, button, or handoff is useful, but it is not proof that code ran, an image was generated, a cron job delivered, a PR passed review, or CI went green.

The point is not that the user knows every backend command. The point is that Hermes can recognize the work shape and return a clear workflow, next action, and evidence state.

Why GEPA-style thinking matters here

Hermes can improve from execution traces, memories, and skills. OMH's job is not to replace that intelligence. OMH should make the traces cleaner: original request, chosen workflow, prepared handoff, observed evidence, user satisfaction, failed claims, and patch candidates.

In other words, Hermes is the capable worker. OMH is the operating layer that makes the work inspectable, routable, reviewable, and improvable over time.

  2. Evaluateschema, install, status, workflow criteria
  3. Reviewhuman approval before skill or workflow mutation
  4. Regressreplay the examples that used to fail

가장 쉬운 이해 방식

보통 챗봇은 한 대화에 답합니다. Hermes Agent는 그보다 작업 운영면에 가깝습니다. 터미널이나 메신저 게이트웨이에서 들어올 수 있고, identity, context, skills, memory, platform hints를 묶어 프롬프트를 만듭니다. 도구를 부르고, 예약 작업을 돌리고, 위임 작업을 만들고, 세션 기록을 남기고, 스킬을 절차적 기억처럼 사용합니다.

그래서 OMH는 "명령어가 더 많은 제품"으로 보이면 약합니다. 진짜 가치는 Hermes의 능력을 더 쉽게 시작하게 하고, 더 안전하게 점검하게 하고, 채팅 안에서 더 명확하게 설명하게 하는 데 있습니다.

요청은 그냥 답변이 되지 않습니다. route가 됩니다.

Hermes를 이해하는 가장 쉬운 방법은 메시지 하나를 따라가보는 것입니다. 사용자는 터미널, Discord, Slack, Telegram, 또는 다른 gateway surface에서 요청합니다. Hermes는 context를 만들고, 관련 skill을 보고, memory나 tool이 필요한지 판단한 뒤 사용자에게 답합니다. OMH는 여기에 제품적인 형태를 붙입니다. 어떤 workflow가 선택됐는지, 다음 action이 무엇인지, 아직 관측되지 않은 것은 무엇인지 보여줍니다.

1

메시지가 들어옴

CLI, TUI, gateway session이 platform, thread, user context를 제공합니다.

2

프롬프트가 구성됨

Identity, skills, memory guidance, context files, platform hints, safety checks가 묶입니다.

3

도구가 실행될 수 있음

Hermes는 허용된 도구를 부르거나 session search, memory, scheduled work 준비를 할 수 있습니다.

4

OMH가 workflow를 이름 붙임

응답은 plan, research, feedback triage, loop, image card, coding handoff를 보여줄 수 있습니다.

5

증거는 나중에 전진함

Status는 wrapper, executor, CI, delivery, review evidence가 기록될 때만 올라갑니다.

01

대화 접점

Hermes는 터미널 UX와 gateway UX를 모두 제공합니다. gateway registry는 built-in 또는 plugin platform adapter가 auth, setup, delivery, privacy, cron delivery behavior를 등록하게 합니다.

02

에이전트 루프

run_agent.py의 AIAgent가 session transition, context engine lifecycle, tool call, response handling, persistence, background review hook을 담당합니다.

03

메모리와 맥락

agent/memory_manager.py는 built-in memory와 최대 하나의 external provider를 함께 조율하고, prefetch, sync, streaming scrub을 통해 recall된 맥락이 최종 응답에 새지 않도록 다룹니다.

04

스킬과 개선 루프

prompt builder는 언제 skill을 저장하거나 고쳐야 하는지 Hermes에 알려줍니다. curator는 agent-created skills를 관리합니다. 즉 OMH가 모델을 학습시키지 않아도, Hermes가 개선 가능한 작업장을 갖게 됩니다.

05

크론과 작업 오케스트레이션

cron/scheduler.py는 lock, protected toolset policy, delivery target, parallel pool, prompt-injection scanning으로 예약 작업을 가드레일 안에서 실행합니다.

06

OMH 계약층

OMH는 Hermes에게 workflow recipe, deterministic routing, status card, coding handoff contract, prepared-vs-observed boundary를 제공해서 wrapper가 추측 없이 상태를 보여주게 합니다.

오해와 실제

오해: OMH가 executor가 되어야 한다.

실제: Hermes와 선택된 executor가 runtime work를 담당합니다. OMH는 route, handoff, status, evidence boundary를 보이게 해야 합니다.

오해: 명령어가 많을수록 제품이 강하다.

실제: 보통 사용자는 Hermes에게 말합니다. 명령어는 installer, verifier, backend, operator infrastructure입니다.

오해: plan은 진행 증거다.

실제: plan은 prepared work입니다. dispatch, implementation, review, CI, delivery는 따로 관측되어야 합니다.

오해: 자기개선은 자동 변경이다.

실제: 더 안전한 루프는 trace, evaluate, propose, review, patch, replay입니다. 사람 리뷰가 skill drift를 막습니다.

OMH는 어디에 붙는가

사용자가 Hermes에게 말함 "이걸 더 안전하게, 명확하게, 출시 가능하게 만들어줘."
Hermes가 생각하고 말함 질문, 리서치, 요약, 계획, 상태 설명.
OMH가 workflow를 형성함 workflow 선택, handoff 준비, local evidence state 기록.
executor 또는 wrapper가 관측함 dispatch, result, review, CI, delivery, merge evidence.

핵심 규칙은 계속 보수적입니다. 준비된 것은 관측된 것이 아닙니다. plan, route, prompt, button, handoff는 유용하지만, 코드가 실행됐거나 이미지가 생성됐거나, 크론이 전달됐거나, PR 리뷰가 통과됐거나, CI가 초록불이라는 증거는 아닙니다.

GEPA식 사고가 여기서 중요한 이유

Hermes는 실행 기록, 메모리, 스킬을 바탕으로 더 나아질 수 있습니다. OMH의 역할은 그 지능을 대체하는 게 아닙니다. OMH는 original request, chosen workflow, prepared handoff, observed evidence, user satisfaction, failed claim, patch candidate 같은 학습 가능한 과정을 더 깨끗하게 남겨야 합니다.

다시 말해 Hermes가 능력 있는 worker라면, OMH는 그 일이 inspectable, routable, reviewable, improvable해지도록 만드는 operating layer입니다.

  1. Tracerequest, route, tool, handoff, evidence
  2. Evaluateschema, install, status, workflow criteria
  3. Reviewskill 또는 workflow 변경 전 사람 승인
  4. Regress전에 실패했던 예제를 다시 돌려보기
